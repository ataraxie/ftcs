Back in the day...

Zero-configuration networking (Zeroconf)

  • Combination of protocols that aim to automatically discover computers or peripherals in a network

Zeroconf

  • Devices are registered with unique IDs (mDNS)

  • Device's services are broadcasted/discovered through the network (DNS-SD)

What if...

  • I want to write one such simple service?

    • Find a device that works for you

    • Write your service and advertise it somehow

    • Write some kind of app for users

What if... (2)

  • I could just implement a Webapp that spins up a server and advertises a server?

  • Then I would need no special device/platform and clients need not download an app

This would...

  • Enable Zeroconf size applications to run natively in the browser

  • Eliminate traditional installation models

  • Enable anyone with a browser becoming a server!

We hypothesize that fault-tolerance is a desirable property for Zeroconf applications that share a state

Successorship

Goals

  • Provide an ease-of-use API to build Zeroconf browser apps

  • Decouple mDNS/DNS-SD logic from the application logic

  • Seamlessly provide fault-tolerance to such apps

Assumptions

  • Clients in the network trust each other

  • Network follows a star topology

  • Applications are not traffic intensive

    • No bursts of requests

    • No payload higher than some threshold Ï„

Successorship Stack

How does it work?

Step-by-step

  • 'App' service is registered under the .local domain

Step-by-step

  • Clients query services in the network

Step-by-step

  • Upon service discovery, client makes a HTTP GET at /

Step-by-step

  • Server serves initial html, js, css files

Step-by-step

  • Client establish a WebSocket connection for further communication

Step-by-step

  • As clients connect, server keeps a list of successors

Step-by-step

  • Clients also keep a replica of the server state

Step-by-step

  • Upon server failure, ws connections are lost

Step-by-step

  • As clients have a replica of the server's last state, they evaluate who is the successor

Step-by-step

  • 'App' service is once again registered under the .local domain

Step-by-step

  • Clients detect a new instance of the same 'App' service and establish a new connection

How do I use it?

API Overview

                
Shippy.register(serviceName, {
    init: function
    operations: map: key-function
});

Shippy.call(operation, payload);

Shippy.on(successorship event, callback);

init

  • Adds app required variables to the server state 
                
let init = function(state) {
    state.queue = [];
};

operations

  • Add app behavior to the server (as operations that manipulate the state)
                
let operations = {
    add: function(state, params) {
        if (state.queue.indexOf(params) < 0) {
            state.queue.push(params);
        }
    }
}

call

  • Calls one of the server operations with a specific payload
                
Shippy.call("add", { name: "Darth Vader", id: "slave-1" });

on

  • Defines how the app will handle server events
                
Shippy.on("stateupdate", function(state) {
    updateUi(state);
});

Under the hood

State broadcast

  • Entire state

  • Operation/payload to be executed on the client end

Handled scenarios

  • Client disconnection

  • Server disconnection

  • Chained disconnection: server and first n> successors
